# 游戏框架(来自队友 Design Prompt,提炼后用于《最后的人》)

来源:队友文档《Claude Design Prompt — AI末日探险直播间游戏界面》(TikTok LIVE 末日生存,观众评论被 AI 采纳生成游戏内容)。
**我们只取它的 UI / 核心交互 / 机制,不取它的故事(丧尸/医院那套)。** 故事用《最后的人》,从原著挖。机制可自由打磨以更好玩、更贴原著。

---

## A. 提炼出的框架(story-agnostic)

**布局(16:9 横屏 · 直播间)**
- 左 70%:游戏主画面
- 右 30%:直播评论区(顶部主播头像+在线人数;评论流;底部输入框+引导语)
- 顶部常驻状态栏:天数进度(Day X/N)| 数值条(多项)| 背包格子

**节奏 / 阶段**
- 以"天/幕"为单位推进(Day 1–N)。
- 每天:**家中**(突发事件 / 整理资源 use 道具)→ **出门**(选目的地 → 探索地图)→ 回家结算。
- 探索:格子地图 + 迷雾 + 行动点(探索消耗)。

**事件类型**
- 搜索类(地点+选项)、NPC 遭遇(头像+对话+交易/招募/情报/离开)、战斗(技能+观众投票票数)、剧情卡(像素插画+打字机文本+继续)。

**资源 / 道具**
- 多维数值条;背包(道具获得/使用/失去,带 toast 与格子闪烁);任何变化弹浮动数字(+绿上浮/−红下沉)。

**观众生成回路(signature,这套的灵魂)**
- 评论 →(每 ~30s 批量)AI 采纳 → 生成游戏内容(事件/道具/NPC/地点)。
- 主播可「征集创意 📢」:倒计时窗口,号召观众投稿。
- 采纳反馈三件套:① 公屏横幅(全场可见,从左滑入)② 自见通知(仅作者可见,星光卡)③ 评论高亮 ✨「已融入游戏」+ 画面光圈汇聚 + 「by @用户」。
- 投票决策:关键抉择/战斗显示实时票数条。
- 结算:**创意排行榜**(最佳创意官/最强脑洞/最佳指挥/最强搅局…)+ 旅程时间线 + 全场统计 + 分享卡。

**双端(演示用)**:主播端操作 → WebSocket 广播 → 看播端音画同步;观众只发评论。Hackathon 最简:Node+WS,评论→AI 可先 mock/调 Claude API。

**里程碑**:胜/败全屏卡(打字机死因/金色粒子)、阶段过场("☀️ Day 4…")、数值浮字、道具 toast、剧情卡暂停。

## B. 接入《最后的人》的决策(主 agent 代为拍板)

1. **美术**:**保留我们的木刻暗调系统 `doom.css`,不走像素**——更贴《最后的人》的文学末世感。但**采纳它的布局与交互范式**(70/30 直播间 + 顶部状态栏 + 评论区 + 采纳反馈三件套 + 投票 + 结算榜)。
2. **资源升到 4 维**(原 3 维 + 1):`食物🥫 / 健康💊 / 士气🔥 / 幸存者👥`。**「幸存者」= 你身边还活着的人数**(伊德丽丝/克拉拉/阿德里安…),随瘟疫一个个失去——极贴原著"最后的人"主题,也比单纯血条更有情感重量。
3. **骨架 = 南逃之旅**:英格兰(温莎)→ 法国 → 瑞士 → 意大利 → 罗马。每一"幕/天"= 旅程一程。「出门/选目的地」= 选路线;「探索」= 沿途地点。这与原著真实地理一致。
4. **事件**:复用我们 `map.json`(66 卡)+ `interactions.json`(197)。"战斗"重皮成**危机/对峙**(瘟疫题材少打斗),用**观众投票决断**;NPC = `codex` 人物;道具/物资 = rich 层的 `stockpile`。
5. **观众回路**:单人模拟观众(可接真 WebSocket)。"采纳一条评论"= 据评论生成下一格的一个事件/道具/地点(先用我们已有卡或轻量现编);采纳反馈三件套照做。
6. **保留并融入**:人物图鉴(点名看背景关系)、手记(我是谁/已遇见的人/历程)、顶栏身份条。
7. **自由度**:机制可改,目标是"更贴原著 + 更好玩",不必逐条照搬。

## C. 要回原著挖的内容(novel mining backlog)
- 南逃旅程各地点的细节(法/瑞/意/罗马沿途场景)→ 探索格子。
- 可成道具/物资的实物(粮、药、船、火、书…)→ 背包。
- 次要人物 → 可招募/遭遇 NPC(扩充 codex)。
- 危机/对峙场景(邪教、哗变、暴雪、沉船…)→ 投票事件。
- 关键剧情卡文本(打字机叙事)。

## D. 沉淀的工作流(可复用到其他小说)
`小说 → world_bible(底座) → map/interactions/characters/codex(内容) → 【新增】framework 适配层(把内容映射进任意游戏壳) → 按壳的机制回小说挖缺口`。
本框架文档 = 适配层的规格;后续换壳/换书,改适配层即可。

---

## E. 通用骨架:住所 + 路线 + POI(v2,适配大多数小说)
"固定基地 + 出门探索"只适合沙盒生存(困守型)小说;多数可爬小说是**旅程/群像**,没有不动的家。改用更通用的三件套:

- **Dwelling(住所,可移动)**:马车 / 营寨 / 房车 / 船 / 临时避难所。有 `type`、`condition`(耐久)、`capacity`、`exposure`(暴露度);会升级 / 受损 / 丢失 / 更换。其脆弱即戏(暴雪夜马车散架→失伊德丽丝)。**cheney 的固定基地 = 不移动的 Dwelling(holdout 特例),完全保留。**
- **Route(路线)**:把章节聚成"程"(legs),每程有地名 / 基调;主线沿此推进。
- **POI(沿途据点)**:每程挂 0~N 个可选据点(废屋 / 营地 / 小镇 / 坠落坑),每个是一张抉择卡 / 际遇,有风险回报;用行动点选去几个。**物资从叙事际遇掉,不是格子刷。**

**spine_type**(由 world_bible 判定)决定地图隐喻与住所是否移动:
| spine_type | 住所 | 地图隐喻 | 例 |
|---|---|---|---|
| journey | 移动 | 旅程图 + 沿途 POI | 最后的人、世界大战 |
| holdout | 固定 | 基地 + 周边探索(= cheney 现有形态) | 困守一地的末世 |
| social | 抽象(你的处境) | 关系网(去见 / 对峙谁) | 群像 / 关系小说 |
| investigation | 据点 | 案件板(线索 / 地点) | 悬疑 |

**三块缝合**:骨架(策展叙事卡 = Route)+ POI(探索 / 物资 / 惊喜)+ 真·AI 观众生成(弹幕生成 POI 及其内容,接 comment_engine)。`poi_bank` = 策展(从小说抽)+ AI 槽(观众生成)。
